2. How The Project Currently Works
============================================================

2.1 Startup and Configuration

The backend starts as a Spring Boot application from:
src/main/java/com/snapfix/SnapfixApplication.java

Scheduling is enabled for token blacklist cleanup.

Runtime configuration is loaded from application.properties and environment variables.
Inside Docker, the backend connects to PostgreSQL using the Docker service hostname:
jdbc:postgresql://postgres:5432/snapfix

PostgreSQL is run with PostGIS through docker-compose.

2.2 Authentication Flow

Register:
POST /auth/register

The user submits email, password, role and name.
The password is BCrypt-hashed.
The user row is created.
Depending on role, either CitizenProfile or WorkerProfile is created.

Login:
POST /auth/login

The backend validates credentials.
It generates:
- Access token, valid for 15 minutes
- Refresh token, valid for 7 days

Existing refresh tokens for the user are revoked during login.
New refresh token is persisted in the refresh_tokens table.

Refresh:
POST /auth/refresh

The endpoint currently accepts the raw refresh token in the request body.
The old refresh token is revoked.
A new access token and refresh token are generated.

Logout:
POST /auth/logout

The refresh token is revoked.
The current access token is added to an in-memory blacklist until its natural expiry.

Known limitation:
The access-token blacklist is in memory. If the backend restarts, blacklisted access tokens become usable again until their normal expiry. This is acceptable for Release 1 but should move to Redis before scaling.

2.3 User Profile Flow

GET /user/me

Returns the authenticated user's:
- userId
- email
- role
- role-specific profile

PUT /user/profile

Citizens can update:
- name
- latitude
- longitude

Workers can update:
- name
- skills

Worker skills are copied into a plain DTO list before serialization to avoid Hibernate lazy loading issues.

2.4 Report Creation Flow

POST /reports

Consumes multipart/form-data.

Required form fields:
- image: file part
- description: text, max 1000 chars
- category: POTHOLE, STREETLIGHT, GARBAGE, WATER_LEAK, ROAD_DAMAGE
- lat: latitude from -90 to 90
- lng: longitude from -180 to 180

Only CITIZEN users can create reports.

Flow:
1. JwtAuthFilter authenticates the request.
2. ReportController extracts multipart data.
3. ReportService validates the image and fields.
4. ReportService searches for nearby reports within 50 metres.
5. If a same-category duplicate exists:
   - No new report row is created.
   - A ReportSupport row is added for the current citizen.
   - supportCount is incremented.
   - The existing report is returned.
6. If no duplicate exists:
   - Image is uploaded through StorageService.
   - Returned Cloudinary URL is stored in imageUrl.
   - A PostGIS Point is created.
   - Report is saved with status CREATED.
   - CitizenProfile.reportsSubmitted is incremented.
   - A ReportSupport row is created for the report creator.
   - supportCount starts at 1.
   - A REPORT_CREATED notification is stored for the creator.

Duplicate report response message:
Existing report found - your support has been added

New report response message:
Report created successfully

2.5 Nearby Reports Flow

GET /reports/nearby?lat={lat}&lng={lng}&radius={radius}

Authenticated users can query nearby reports.
Default radius is 5000 metres.

The query uses:
- ST_DWithin for radius filtering
- ST_Distance for sorting
- location::geography casts for accurate metre-based distance

Important:
Longitude must be passed first to ST_MakePoint.
GeoUtil.createPoint(lat, lng) stores coordinates as Coordinate(lng, lat).

2.6 Support Flow

POST /reports/{id}/support

Only CITIZEN users can support reports.

Flow:
1. Report is loaded by id.
2. Current user id is resolved from CustomUserDetails.
3. The service checks whether this user already supported the report.
4. If not supported:
   - ReportSupport row is created.
   - supportCount is incremented.
   - A REPORT_SUPPORTED notification is stored for the original report owner.
5. If already supported:
   - 400 Bad Request is returned.

Database constraint:
report_support has a unique constraint on (report_id, user_id).

2.7 Notification Flow

GET /notifications

Authenticated users can retrieve their own notifications.
The optional unread query parameter filters by read state:
- unread=true returns unread notifications
- unread=false returns read notifications
- omitted returns all notifications for the current user

PATCH /notifications/{id}/read

Marks the current user's notification as read.
Users cannot mark another user's notification as read; the service returns 404 for missing or foreign notifications.

Stored notification types currently used in Release 1:
- REPORT_CREATED
- REPORT_SUPPORTED

2.8 Worker Discovery Flow

GET /workers/reports/nearby?lat={lat}&lng={lng}

Only WORKER users can access this endpoint.
It returns nearby reports within 5 kilometres using the existing report geo-query path.

4. Bugs and Issues Encountered
============================================================

Issue 1: Invalid JWT returned 403 instead of 401

Root cause:
JwtAuthFilter continued the filter chain after detecting invalid tokens.

Impact:
Spring Security treated the request as access denied instead of unauthenticated.

Fix:
JwtAuthFilter now writes a 401 response directly and returns immediately.

Status:
Fixed.

------------------------------------------------------------

Issue 2: Logout returned 500 in some invalid-token cases

Root cause:
Transactional exception behavior and token revocation paths were not mapped cleanly.

Impact:
Client received raw 500 instead of controlled error.

Fix:
Invalid logout state now uses typed exceptions. Refresh tokens are explicitly saved after revocation.

Status:
Fixed.

------------------------------------------------------------

Issue 3: application.properties used YAML-style syntax earlier

Root cause:
Properties file had nested YAML-like keys.

Impact:
JWT secret could resolve as null in Docker, breaking token generation.

Fix:
All properties must use flat dot notation.

Status:
Fixed.

------------------------------------------------------------

Issue 4: Backend could start before PostgreSQL was ready

Root cause:
Docker Compose depends_on only waited for container startup, not DB readiness.

Impact:
Backend could fail during startup.

Fix:
PostgreSQL healthcheck and service_healthy dependency were added.

Status:
Fixed for Docker Compose.

------------------------------------------------------------

Issue 5: Access token remained valid after logout

Root cause:
JWT access tokens are stateless.

Impact:
Logout revoked refresh token but not current access token.

Fix:
In-memory token blacklist added.
JwtAuthFilter checks blacklist before authentication.
Cleanup task removes expired blacklist entries.

Status:
Fixed for single-instance Release 1.

Remaining risk:
Blacklist resets on restart and does not work across multiple backend instances.
Move to Redis in a later release.

------------------------------------------------------------

Issue 6: Testcontainers failed on Windows Docker Desktop earlier

Root cause:
Docker daemon discovery was not configured correctly.

Impact:
Integration tests could not start PostgreSQL containers.

Fix/workaround:
Docker Desktop must be running and Testcontainers must be configured to use the available Docker pipe.
Current run successfully detected:
npipe:////./pipe/docker_engine

Status:
Resolved in current environment.

------------------------------------------------------------

Issue 7: Spring Boot test mocking package changed

Root cause:
Old @MockBean package is deprecated/removed in newer Spring Boot/Spring Framework test stack.

Impact:
Compilation failures when using old mocking annotation.

Fix:
Use @MockitoBean from:
org.springframework.test.context.bean.override.mockito.MockitoBean

Status:
Fixed.

------------------------------------------------------------

Issue 8: RestAssured crashed before sending requests

Root cause:
RestAssured proxy setup failed with:
Cannot invoke "Object.hashCode()" because "key" is null

Impact:
Security integration tests failed before reaching the backend.

Fix:
Integration tests now use Java's built-in HttpClient.

Status:
Fixed.

------------------------------------------------------------

Issue 9: Testcontainers container stopped while Spring context was reused

Root cause:
Each integration class had a separate container lifecycle, but Spring reused an app context whose datasource pointed to the first stopped container.

Impact:
Later DB tests failed with connection refused.

Fix:
BaseIntegrationTest starts a singleton PostGIS container for the test JVM.

Status:
Fixed.

------------------------------------------------------------

Issue 10: Duplicate refresh JWTs generated in same second

Root cause:
JWT contents were identical when login and refresh happened in the same second for the same subject.

Impact:
New refresh token could violate unique constraint in refresh_tokens.token.

Fix:
JwtUtil now adds a random JWT ID claim, jti.

Status:
Fixed.

------------------------------------------------------------

Issue 11: refresh_tokens.token was too short

Root cause:
Default column length was varchar(255).

Impact:
JWT with jti could exceed 255 characters and fail insert.

Fix:
RefreshToken.token column length increased to 1024.

Status:
Fixed.

------------------------------------------------------------

Issue 12: WorkerProfile skills caused lazy serialization failure

Root cause:
WorkerProfile.skills is an ElementCollection and could remain lazy when serialized.

Impact:
GET /user/me for workers could fail with LazyInitializationException.

Fix:
UserService maps skills into a new ArrayList inside a transaction.

Status:
Fixed.

Fixed.

6. Current Remaining Risks
============================================================

1. Secrets are present in application.properties.
   Move Cloudinary and JWT values to environment variables only.

2. Hibernate ddl-auto is still being used.
   Add Flyway/Liquibase before schema becomes more complex.

3. In-memory access-token blacklist is not production-grade.
   Use Redis later.

4. Notifications are persisted and retrievable, but there is no real-time push/WebSocket delivery yet.
   Real-time delivery belongs in a later release.

5. reports.location does not yet have an explicit spatial GiST index.
   Current performance test passes at 1,000 reports, but production should use a spatial index.

6. Report.citizenId and ReportSupport IDs are UUID fields, not JPA relationships.
   This keeps modules simpler, but DB-level referential integrity is incomplete.

7. Report status lifecycle is only an enum today.
   No workflow exists yet for IN_PROGRESS or COMPLETED.

8. Docker Compose smoke test has not been recorded after Phase 4.
   Run docker-compose up --build and verify /actuator/health plus a basic auth/report flow.
