2. Implemented Release 3 Phase 1 and 2 Scope
============================================================

2.1 Proof of Work Upload

Implemented endpoints:
- POST /tasks/{taskId}/proof
- GET /tasks/{taskId}/proof

Behavior:
- Only the assigned worker can upload proof.
- Upload requires task status IN_PROGRESS.
- Proof stores Cloudinary image URL, PostGIS GPS point, remarks, task and worker.
- Upload moves the task from IN_PROGRESS to PROOF_SUBMITTED.
- One proof per task is enforced.
- Proof viewing is limited to assigned worker, report citizen or admin.

2.2 Citizen Verification

Implemented endpoint:
- POST /tasks/{taskId}/verify?status=VERIFIED|REJECTED

Behavior:
- Only the citizen who created the original report can verify/reject proof.
- Verification is allowed only when the task is PROOF_SUBMITTED.
- VERIFIED moves the task to VERIFIED_BY_CITIZEN.
- REJECTED moves the task to REJECTED.
- Citizen rejection does not increment retryCount.
- Verification records store task id, citizen id, status, comments and timestamp.

Learning note:
The useful domain distinction is that a rejection is a citizen decision, while a
retry is a worker action. retryCount should count actual worker retry attempts,
not the number of times a citizen rejected work.

3. Implemented Release 3 Phase 3 and 4 Scope
============================================================

3.1 Worker Retry

Implemented endpoint:
- POST /tasks/{taskId}/retry

Behavior:
- Only the assigned worker can retry their task.
- Retry is allowed only from REJECTED.
- Retry moves the task from REJECTED back to IN_PROGRESS.
- Retry increments Task.retryCount.
- Retry is blocked when retryCount is already 3 or more.

3.2 Admin Final Review and Reassignment

Implemented endpoints:
- GET /admin/tasks?status={TaskStatus}
- GET /admin/tasks/{id}
- POST /admin/tasks/{taskId}/approve
- POST /admin/tasks/{taskId}/reject
- POST /admin/tasks/{taskId}/reassign

Behavior:
- Admin can list tasks, optionally filtered by status.
- Admin can view task detail composed of TaskResponse, ProofResponse and
  VerificationResponse.
- Admin approval requires task status VERIFIED_BY_CITIZEN.
- Admin approval moves the task to COMPLETED and the report to COMPLETED.
- Admin rejection requires task status VERIFIED_BY_CITIZEN and moves the task to
  REJECTED.
- Admin reassignment requires a newWorkerId request body.
- Reassignment blocks completed tasks and completed reports.
- Reassignment sets the new worker, resets the task to ASSIGNED and updates
  assignedAt.
- Admin task decisions are written to AdminActionLog.

4. Release 2 Bugs and Fixes
============================================================

Issue 21: Admin approval called bid approval twice

Root cause:
AdminService initially called BidService.approveBid and then repeated state transitions.

Impact:
The second call could fail because the bid was already APPROVED and the report was already IN_PROGRESS.

Fix:
AdminService calls BidService.approveBid once, then logs the returned approved bid.

Status:
Fixed.

------------------------------------------------------------

Issue 22: Admin rejection called bid rejection twice

Root cause:
AdminService called BidService.rejectBid twice.

Impact:
The first call changed the bid to REJECTED; the second call failed because the bid was no longer ACTIVE.

Fix:
AdminService calls BidService.rejectBid once, then logs the returned rejected bid.

Status:
Fixed.

------------------------------------------------------------

Issue 23: Task endpoints allowed worker role but needed ownership checks

Root cause:
hasRole('WORKER') confirms the user is a worker, but not that the task belongs to that worker.

Impact:
A worker who knew another task UUID could fetch or start that task.

Fix:
TaskService checks task.worker.id against the authenticated worker id.

Status:
Fixed.

------------------------------------------------------------

Issue 24: TaskResponse exposed entities

Root cause:
TaskResponse originally returned Report and User objects.

Impact:
DTO boundary was violated and entity internals could leak.

Fix:
TaskResponse returns reportId and workerId UUIDs.

Status:
Fixed.

------------------------------------------------------------

Issue 25: Worker location used embedded latitude/longitude

Root cause:
WorkerProfile.currentLocation initially used the shared Location embeddable.

Impact:
Release 2 required worker currentLocation to be PostGIS geometry.

Fix:
WorkerProfile.currentLocation now uses JTS Point with Hibernate Spatial.

Status:
Fixed.

------------------------------------------------------------

Issue 26: Worker nearby discovery accepted arbitrary query coordinates

Root cause:
Endpoint accepted lat/lng query params.

Impact:
The behavior did not match the Developer Reference, which says to use worker current location.

Fix:
GET /workers/reports/nearby now uses WorkerProfile.currentLocation.

Status:
Fixed.

------------------------------------------------------------

Issue 27: Registration-created worker profile conflicted with POST /workers/profile

Root cause:
Auth registration creates a basic worker profile. Later POST /workers/profile saw the existing row
and returned "already exists".

Impact:
Workers could not complete their Release 2 profile through the intended endpoint.

Fix:
POST /workers/profile completes an existing profile when currentLocation is still missing.

Status:
Fixed.

------------------------------------------------------------

Issue 28: resourceNote optional field was modeled incorrectly

Root cause:
BidRequestDTO used Optional<String> and validation logic treated present values incorrectly.

Impact:
Valid bids could fail validation, and missing resourceNote was awkward to handle.

Fix:
BidRequestDTO uses a normal nullable/default String. Service stores empty string when omitted.

Status:
Fixed.

------------------------------------------------------------

Issue 29: Task route used singular path

Root cause:
Controller used /task.

Impact:
Developer Reference specifies /tasks.

Fix:
TaskController now uses /tasks.

Status:
Fixed.

------------------------------------------------------------

Issue 30: Task retryCount default was 3

Root cause:
retryCount was initialized as if it represented remaining retries.

Impact:
Release 3 retry logic expects attempts used, starting at 0.

Fix:
Task.retryCount defaults to 0.

Status:
Fixed.

------------------------------------------------------------

Issue 31: Report response lat/lng mapping was reversed in one constructor

Root cause:
JTS Point x/y was confused with lat/lng.

Impact:
Admin report listing could return swapped coordinates.

Fix:
ReportResponse maps lat from point.getY() and lng from point.getX().

Status:
Fixed.

Issue 32: Missing proof image returned generic 500

Root cause:
Missing multipart request parts were handled by the generic exception path.

Impact:
Client-side request mistakes looked like server failures.

Fix:
Map missing multipart request parts to HTTP 400.

Learning:
Multipart endpoints need explicit error handling because framework-level binding
errors happen before service validation.

Status:
Fixed.

------------------------------------------------------------

Issue 33: Proof access had role checks but not ownership checks

Root cause:
The endpoint checked authenticated roles but did not confirm relationship to the
task/report.

Impact:
An authenticated user could potentially view proof by task id.

Fix:
Allow only the assigned worker, original report citizen or admin.

Learning:
Role-based authorization is necessary but not sufficient for object-level data.

Status:
Fixed.

------------------------------------------------------------

Issue 34: Citizen verification used worker-owned task lookup

Root cause:
Verification reused a task lookup that enforced assigned-worker ownership.

Impact:
The report citizen could be blocked from verifying their own report's task.

Fix:
Use a neutral task lookup for citizen-owned verification, then perform citizen
ownership checks separately.

Learning:
Shared service methods should make their authorization assumptions obvious in
the method name.

Status:
Fixed.

------------------------------------------------------------

Issue 35: Verification mutated task status before validating source status

Root cause:
The service changed task state before checking that the task was
PROOF_SUBMITTED.

Impact:
Valid lifecycle checks could fail after mutation, and invalid transitions became
harder to reason about.

Fix:
Validate source state first, then mutate state.

Learning:
State machines should follow this order: load, authorize, validate source
state, validate command, mutate, persist.

Status:
Fixed.

------------------------------------------------------------

Issue 36: Verification stored the wrong citizen id

Root cause:
The task id was accidentally stored where the citizen id belonged.

Impact:
Verification audit records were misleading.

Fix:
Store the original report owner's citizen id.

Learning:
Audit records deserve tests for identity fields, not only status fields.

Status:
Fixed.

------------------------------------------------------------

Issue 37: Rejection with comments could verify a task

Root cause:
Branching logic depended on comments/status handling instead of the explicit
requested verification status.

Impact:
A rejected proof could be recorded as verified in some paths.

Fix:
Branch explicitly on VERIFIED and REJECTED.

Learning:
For enum-driven commands, branch on the enum, not on secondary fields such as
comments.

Status:
Fixed.

------------------------------------------------------------

Issue 38: Retry count was coupled to citizen rejection

Root cause:
Earlier logic incremented retryCount when the citizen rejected proof.

Impact:
retryCount represented rejections rather than worker retry attempts.

Fix:
Citizen rejection now only moves the task to REJECTED. POST /tasks/{taskId}/retry
increments retryCount when the worker actually retries.

Learning:
Name counters after the event they count. If the field is retryCount, increment
it on retry, not on rejection.

Status:
Fixed.

------------------------------------------------------------

Issue 39: Retry needed a hard limit at the retry action

Root cause:
The max retry rule was originally enforced around rejection semantics, not the
retry command itself.

Impact:
The fourth retry attempt needed direct protection.

Fix:
TaskService.retryTask blocks retry when retryCount >= 3 and returns conflict
through the global exception mapping.

Learning:
Limits should be enforced at the command that consumes the limit.

Status:
Fixed.

------------------------------------------------------------

Issue 40: Admin task detail needed a neutral aggregate response

Root cause:
Admin review needs task, proof and verification together, while worker task
lookup enforces assigned-worker ownership.

Impact:
Admin final review could not safely reuse worker-specific lookup paths.

Fix:
Add TaskDetail DTO and compose TaskResponse, ProofResponse and
VerificationResponse in AdminService.

Learning:
Admin read models often differ from worker/citizen read models. Keep them DTO
based and avoid exposing entities.

Status:
Fixed.

------------------------------------------------------------

Issue 41: Admin final approval needed lifecycle guards

Root cause:
Admin approval is a terminal transition, so it needed strict source-state
validation.

Impact:
Approving before citizen verification would skip a required workflow gate.

Fix:
Allow admin approval only when task status is VERIFIED_BY_CITIZEN and report
status is IN_PROGRESS.

Learning:
Terminal transitions should be the most defensive transitions in the workflow.

Status:
Fixed.

------------------------------------------------------------

Issue 42: Reassignment needed terminal-state protection

Root cause:
Reassignment changes worker ownership and status, so completed work must be
protected.

Impact:
Completed tasks or completed reports could be reopened incorrectly.

Fix:
Block reassignment for COMPLETED/PAYMENT_RELEASED tasks and COMPLETED reports.

Learning:
Before ownership reassignment, check both the task state and the parent report
state.

Status:
Fixed.

Issue 43: Local full verification blocked by Docker/Testcontainers

Issue 44: Payment release needs duplicate-credit protection

Root cause:
Payment release is a money-moving command. If it can be repeated after a
payment is released, worker wallets can be credited more than once for the same
task.

Impact:
Wallet balances and transaction history would become incorrect.

Fix:
Release is allowed only while payment status is PENDING. Duplicate release
attempts return conflict and do not create extra transactions or wallet credit.

Learning:
Money-moving endpoints should validate source state before mutating wallet,
payment, task or transaction records.

Status:
Verified by Release 3 Phase 5 tests.

------------------------------------------------------------

Issue 45: Ratings need task-level uniqueness and lifecycle guards

Root cause:
Ratings affect worker averages and should represent completed work, not any
arbitrary worker/citizen interaction.

Impact:
Duplicate or premature ratings could distort worker reputation.

Fix:
Only the report citizen can rate the worker for completed work, and one rating
per task is enforced.

Learning:
Reputation data should be tied to a completed domain event and guarded like an
audit record.

Status:
Verified by Release 3 Phase 6 tests.
